# Work OS — Blueprint Runtime v1 (PR #75)

The first concrete execution runtime for Loop Work OS. It makes the execution
loop real: **create work → assign owner → complete stage → assign/notify next
owner → next owner sees it.**

## Vocabulary

- **Blueprint** — a reusable process template.
- **BlueprintStage** — one step in a template (has an optional default owner).
- **WorkInstance** — a real execution of a Blueprint.
- **WorkStage** — one step of a real execution (copied from a BlueprintStage).
- **WorkAssignment** — ownership record for a stage (audit of who owned what).
- **WorkNotification** — in-app notification telling the next owner it is their turn.
- **WorkComment** — discussion on a work instance / stage.

> Company permission (owner / admin / employee) is **separate** from project
> responsibility. People can own different stages depending on the work. No
> employees are hardcoded into any workflow.

## Runtime rules

**Create work from a blueprint**

1. Copy the BlueprintStages into WorkStages, preserving order.
2. First stage → `ready`; all others → `pending`.
3. `currentStageId` = first WorkStage id.
4. First owner = first stage's default owner if set, otherwise the explicit
   owner chosen by the creator.
5. If a first owner exists, create a `next_action_ready` notification:
   *"Your next action is ready — {title}: {stage}"*.

**Complete the current stage**

1. Mark the current stage `completed` with `completedAt` + `completedByUserId`.
2. Find the next stage by position.
3. No next stage → mark the WorkInstance `completed`, set `completedAt`, clear `currentStageId`.
4. Next stage exists → set it `ready`; owner = provided next owner, else the copied
   default owner, else `null`; move `currentStageId` to it.
5. If the next stage has an owner, notify them:
   *"{Actor} completed {previous stage}. Your next step is {next stage}."*
6. If no next owner is chosen and none was copied, the stage is `ready` but
   **unassigned** — it appears in the admin queue as **"Needs owner"**.

## Seed / example blueprint — "New Buyer Onboarding"

Create this via **/app/admin/work/blueprints/new** (it is pre-filled as the
default stage list):

1. Review Agreement
2. Create Buyer Record
3. Campaign Setup
4. QA Campaign
5. Go Live

EMG can model the same way (all on the one Blueprint system): **New Buyer,
New Vendor / Source, New Content Creator, New Business Owner, Website Project,
Government Proposal, Marketplace Investigation.**

## Walkthrough scenario (Charlie → Mike)

1. Create the blueprint **New Buyer Onboarding**.
2. Create a work instance **"New Buyer — ABC Roofing"** from it.
3. Stage 1 (Review Agreement) is assigned to **Charlie** and he is notified.
4. Charlie completes Stage 1 and picks **Mike** as the next owner.
5. Stage 2 (Create Buyer Record) becomes `ready`, owned by Mike.
6. Mike receives an in-app notification:
   *"Charlie completed Review Agreement. Your next step is Create Buyer Record."*
7. Mike sees Stage 2 in **My queue** on /app/admin/work.

## Surfaces (v1)

- `/app/admin/work` — Next action, My queue, Waiting / Unassigned, Completed today, Notifications.
- `/app/admin/work/new` — create a work instance from a blueprint.
- `/app/admin/work/[id]` — work detail: stages, owners, complete-stage, reassignment, comments.
- `/app/admin/work/blueprints` + `/blueprints/new` — manage blueprints.

## Scope & tradeoffs (v1)

- **Admin-only UI.** These routes run behind `requireWorkspace('ADMIN')`, serving
  the owner/admin workspace (Matt, Charlie, Jonathan, Mike). A dedicated
  employee-workspace queue (Francesca, Alex, Brian) is a **follow-up PR**; the
  backend queue methods already support per-user queues.
- **In-app notifications only.** No email / Slack / SMS.
- **Scalar links.** New models use scalar `organizationId` / `userId` columns
  (no Prisma relations back onto Organization / User) to keep the migration
  purely additive. Promoting these to full relations is future work.
- **Hand-written migration.** The migration under
  `packages/database/prisma/migrations` was authored by hand to match the schema
  additions; it was **not** generated by `prisma migrate dev`. Run
  `prisma migrate diff` / `prisma migrate dev` in an environment with DB access
  to validate before applying.
- **Status vocabulary** is stored as string scalars (e.g. `active`, `ready`,
  `in_progress`, `next_action_ready`) matching the spec exactly, enforced in the
  service layer rather than as Prisma enums.
- The existing types-only `packages/work-os` contracts package is left as-is;
  reconciling its vocabulary with this concrete runtime is future work.
